# M0.3 主持人设计

> 版本：v0.1（待审）
> 上游：M0.1 Router「困境卡片」 + M0.2 三 agent 输出
> 下游：M0.4 编排脚本（实施）+ 给用户的最终回复

---

## 一、主持人的本质：不是第 4 个角色

主持人**不是另一个 persona agent**——他没有人格、没有立场、没有语料库。

主持人是**两个产品身份的合成**：

| 身份 | 职责 |
|---|---|
| **编排控制器** | 决定谁说话、说几次、按什么顺序、跳过谁 |
| **综合输出员** | 把 3 agent 输出捏成给用户友好的最终回复 |

主持人的"声音"只有一种：**温暖但克制的中性产品声音**——像一个圆桌讨论的引导者，不抢戏、不站队。

### 1.1 为什么主持人不能有人格？
- 已经有 3 个角色 agent 在表达立场了，再加一个会让用户被 4 个声音轰炸
- 主持人的核心价值是"让用户在 3 人观点之间自己选"，不是"再多一个观点"
- 中性立场让兜底场景更优雅（某 agent 输出失败时主持人能无缝补位）

### 1.2 主持人的 4 种独立任务
1. **追问触发**：信息不齐时，先用 Router 给的 followup_questions 问用户，不调用 3 agent
2. **危机接管**：Router crisis=true 时跳过 3 agent，主持人独立处理
3. **3 人调度**：根据 intent 决定谁先发、要不要互驳、要不要某人让位
4. **综合输出**：把 3 人发言+互驳织成一个完整回复

## 二、整体编排流程

```
用户输入
  ↓
M0.1 Router → 困境卡片 (YAML)
  ↓
┌──────────────── 主持人接管 ────────────────┐
│                                              │
│  Step 1：扫描卡片                            │
│   ├─ crisis=true → 走【危机分支】 → 终止    │
│   ├─ needs_followup=true → 走【追问分支】   │
│   │     → 拿到用户回答 → 回到 M0.1 重判      │
│   └─ 否则 → 走【主流程】                    │
│                                              │
│  Step 2：根据 intent 分流（主流程）         │
│   ├─ intent=listen   → 苏轼首发模式         │
│   ├─ intent=decision → 三人并行首发模式     │
│   └─ intent=advice   → 三人并行首发模式     │
│                                              │
│  Step 3：2 轮深度辩论（仅 decision/advice） │
│   ├─ 轮次 0  3 人首发（并行）                │
│   ├─ 轮次 1  反驳 + 回应（双 stage 各并行） │
│   ├─ 轮次 2  再驳 + 再回应（同上）          │
│   └─ 每人发言 5 次，驳 2 个 peer，被 2 驳   │
│                                              │
│  Step 4：handover/mirror 特殊处理           │
│   ├─ handover_to_peer → 改写第 2 轮反驳     │
│   └─ mirror_to_peer → 综合输出时高亮         │
│                                              │
│  Step 5：综合输出织文                       │
│   └─ 按结构化模板生成最终回复               │
│                                              │
└──────────────────────────────────────────────┘
  ↓
返回给用户
```

## 三、Step 1：危机分支

### 3.1 触发
Router 输出 `crisis: true`。

### 3.2 主持人响应（**完全跳过 3 agent**）
```
看到你说的话，我们停一停。

刚才你说的【XX】（直接引用 crisis_reason 里的具体词），
不像是只是工作的事。

如果你只是太累了，可以告诉我，我们慢慢来。
但如果你有伤害自己的念头，请现在就拨打：
- 全国心理援助热线：400-161-9995
- 北京心理危机研究与干预中心：010-82951332

不论你现在是什么状态，先放下手机，到家人或朋友身边。
我们之后再谈工作的事。
```

### 3.3 关键设计决策
- **不让 3 agent 发言** —— 避免任何"安慰话术"被理解为对生死议题的轻慢
- **直接引用用户原话** —— 让用户感到"被听见"，而不是模板回复
- **明确转介**至专业资源 —— 法律/伦理底线
- **不"诊断"** —— 不说"你是抑郁症"、不评判
- **保留对话空间** —— "可以告诉我，我们慢慢来"

### 3.4 输出 schema
```yaml
output_type: crisis_response
crisis_reason_quoted: <Router 给的 crisis_reason>
text: <按上述模板填充的话>
followup_offered: false  # 危机时不追问
agents_consulted: []     # 3 agent 全部跳过
emergency_resources_listed: true
```

## 四、Step 1.5：追问分支

### 4.1 触发
Router `needs_followup: true` 且 `crisis: false`。

### 4.2 主持人响应（**完全跳过 3 agent**）
直接把 Router 给的 `followup_questions` 用主持人语气包装：

```
你说的【一句用户原话片段】，我们三个想再听清楚一件事——

{followup_questions[0]}
（如果有 questions[1]，也一起问，但最多 2 个，避免审讯感）

不必详细，你方便说多少都行。
```

### 4.3 等用户回答后的处理
```
用户回答 → 拼接到原始输入
       → 完整文本重新走 M0.1 Router
       → 新一轮判定（一般 needs_followup 应变 false）
       → 进入主流程
```

最多追问 2 轮，第 2 轮后即使信息不齐也强制进入主流程，主持人在综合时主动声明"信息有限"。

### 4.4 输出 schema
```yaml
output_type: followup_question
text: <追问的话>
questions: <来自 Router 的 followup_questions>
followup_round: <1 或 2>
agents_consulted: []
```

## 五、Step 2:主流程 - 调度策略

### 5.1 按 intent 分流的依据

| intent | 含义 | 调度策略 | 互驳？ |
|---|---|---|---|
| **listen** | 用户在倾诉、宣泄 | **苏轼首发模式** | ❌ 跳过互驳 |
| **decision** | 用户在选 A 还是 B | **三人并行首发** | ✅ 进入互驳 |
| **advice** | 用户问"怎么办"未决 | **三人并行首发** | ✅ 进入互驳 |

### 5.2 苏轼首发模式（intent=listen）

设计意图：用户在倾诉时最不需要被点醒/被压日课。先让苏轼抚情绪，曾国藩/王阳明克制补一句即可。

```
调度顺序：
1. 苏轼 first_opinion（完整发言，500-800 字）
2. 曾国藩 first_opinion（克制版，100-150 字）
   — 提示词追加："此用户在倾诉。请只补一句'事在人为'式的话，
                  不必给日课、不必反问。"
3. 王阳明 first_opinion（克制版，80-120 字）
   — 提示词追加："此用户在倾诉。请克制点醒冲动。
                  最多补一句'你哭出来是好事'式的共情，
                  不抛 mirror_question。"

互驳：跳过
```

### 5.3 三人并行首发模式（intent=decision/advice）

3 个 agent 并行调用 LLM，等全部回来再进入互驳。

```python
# 伪代码
results = parallel_call([
    曾国藩.first_opinion(router_card),
    苏轼.first_opinion(router_card),
    王阳明.first_opinion(router_card),
])
```

### 5.4 调度决策矩阵

| Router 字段 | 影响 |
|---|---|
| `intent` | 决定首发模式（见 5.1） |
| `score` | < 30 时即使没 needs_followup 也提示主持人在综合时声明"信息有限" |
| `category` | 桶 4（裁员/被劝退）+ 桶 7（倦怠）→ 苏轼权重提高 |
| `category` | 桶 1（跳槽）+ 桶 5（35 焦虑）→ 王阳明权重提高 |
| `emotional_keywords` | 含"急/慌/睡不好" → 苏轼优先抚情绪句 |
| `core.user_already_knows` | true → 王阳明权重提高（直击自欺最有效） |

权重影响的不是发言顺序，而是**综合输出时该 agent 的引用比重**（见 Step 5）。

## 六、Step 3：2 轮深度辩论（仅 decision/advice）

### 6.1 设计原则：让 agent 真吵起来
3 人辩论的产品价值不是"3 段独白+互怼"，而是**完整的「立 → 驳 → 应 → 再驳 → 再应」闭环**。
每个 agent 在 2 轮辩论里发言 5 次：
1. 首发（Stage 1）
2. 反驳一个 peer（Stage 2，第 1 轮）
3. 被另一个 peer 反驳后回应（Stage 3，第 1 轮）
4. 反驳另一个 peer（Stage 2，第 2 轮）
5. 被又一个 peer 反驳后回应（Stage 3，第 2 轮）

每个 agent 在 2 轮里**驳过 2 个 peer，被 2 个 peer 驳过**——三角全方位质询。

### 6.2 配对设计：双闭环换边

**第 1 轮配对（驳→应顺时针闭环）**

| 驳论方 → 被驳方 | 第 1 轮戏剧张力 |
|---|---|
| 曾国藩 → 苏轼 | 苦修 vs 旷达 |
| 苏轼 → 王阳明 | 安住 vs 点醒 |
| 王阳明 → 曾国藩 | 心法 vs 事功 |

**第 2 轮配对（驳→应逆时针闭环）**

| 驳论方 → 被驳方 | 第 2 轮戏剧张力 |
|---|---|
| 曾国藩 → 王阳明 | 日课的实效 vs 良知的虚空 |
| 王阳明 → 苏轼 | 旷达是真安还是麻醉 |
| 苏轼 → 曾国藩 | 笨功夫的边界 vs 人之为人 |

**为什么用换边而不是连续追打同一对？**
- 换边：每个 agent 立场被 3 人各自质询过（话题面广）
- 追打：A↔B 连续吵 4 句话（深度 vs 重复风险）
- 选换边的核心理由：**第 2 轮反驳能引用对手的 Stage 3 回应**——形成"你刚才让步了，我借你松口的地方推进"的真实辩论感。这是连续追打做不到的。

### 6.3 完整调度时序

```
轮次 0（首发）：
  Step 3.0  3 人并行首发
            曾国藩.first_opinion()
            苏轼.first_opinion()
            王阳明.first_opinion()
            ────────  3 次 LLM，~5s

轮次 1（驳 + 应）：
  Step 3.1  3 人并行反驳（看 peer 的首发）
            曾国藩.rebuttal(target=苏轼,    peer=苏轼.first)
            苏轼.rebuttal(target=王阳明,    peer=王阳明.first)
            王阳明.rebuttal(target=曾国藩,  peer=曾国藩.first)
            ────────  3 次 LLM，~5s

  Step 3.2  3 人并行回应（看自己被谁驳了）
            苏轼.adjust(round=1, trigger=曾国藩.rebuttal_at_苏)
            王阳明.adjust(round=1, trigger=苏轼.rebuttal_at_王)
            曾国藩.adjust(round=1, trigger=王阳明.rebuttal_at_曾)
            ────────  3 次 LLM，~5s

轮次 2（再驳 + 再应）：
  Step 3.3  3 人并行再反驳（驳另一个 peer，且引用对方的 Stage 3 回应）
            曾国藩.rebuttal(target=王阳明,  peer=王阳明.first + 王阳明.adjust_round1)
            王阳明.rebuttal(target=苏轼,    peer=苏轼.first + 苏轼.adjust_round1)
            苏轼.rebuttal(target=曾国藩,    peer=曾国藩.first + 曾国藩.adjust_round1)
            ────────  3 次 LLM，~5s

  Step 3.4  3 人并行再回应
            王阳明.adjust(round=2, trigger=曾国藩.rebuttal2_at_王)
            苏轼.adjust(round=2, trigger=王阳明.rebuttal2_at_苏)
            曾国藩.adjust(round=2, trigger=苏轼.rebuttal2_at_曾)
            ────────  3 次 LLM，~5s
```

**总计：15 次 LLM 调用，5 个 stage 串行（每 stage 内 3 次并行），~25s**

### 6.4 关键设计决策

| 决策点 | 选择 | 理由 |
|---|---|---|
| 同 stage 内 3 次调用 | 并行 | 无依赖，省时间 |
| 跨 stage（首发→反驳→回应） | 串行 | 必须看到上一 stage 输出 |
| 第 2 轮反驳能否引用对方 Stage 3 | **能** | 这是 2 轮的核心戏剧机制 |
| Stage 3 回应能否再追问 | **不能** | 否则无限循环。坚定让 agent 在 2 轮内打住 |
| handover_to_peer 在哪一轮处理 | 第 2 轮替换该 agent 的反驳目标 | 见 §7 |
| mirror_to_peer | 不触发新调用，主持人在综合输出时高亮 | 见 §7 |

### 6.5 失败时的退化策略

| 失败位置 | 处理方式 |
|---|---|
| 轮次 0 首发某人失败 | 该 agent 整局退出，2 轮辩论降级为 2 人辩 |
| 轮次 1 反驳失败 | 该方向跳过，对应 Stage 3 回应也跳过 |
| 轮次 1 回应失败 | 不影响第 2 轮反驳（用 Stage 1 兜底） |
| 轮次 2 反驳失败 | 该方向跳过，回应也跳过 |
| 任何全失败 | 触发 §9 多 agent 失败兜底 |

### 6.6 driver_pair 是否动态？
v0.2 决策：**配对固定**——双闭环换边的设计已经覆盖所有场景。
- 即使 category=情绪类（如 PUA），第 1 轮苏轼→王阳明的"安住 vs 点醒"也能切到情绪治理
- 即使 category=决策类（如跳槽），第 2 轮曾国藩→王阳明的"日课实效 vs 良知虚空"也能切到决策犹豫

M0.5 实测发现某场景配对不合适，再加 override 规则。当前不预设。

## 七、Step 4：handover / mirror 特殊处理

2 轮辩论是默认结构，但 3 个 agent 的输出里有两类**跨越流程的特殊字段**，需要主持人单独处理：

### 7.1 苏轼的 handover_to_peer
苏轼遇到结构性问题（薪资机制、绩效流程等）时，会主动让位：
```yaml
handover_to_peer:
  peer: 曾国藩  # 或 王阳明
  reason: "此涉职场实务，吾未深谙，可问涤生兄"
```

**主持人处理**：
- 该字段非空时，**第 2 轮反驳改写规则**：原本苏轼应该驳曾国藩，改成苏轼**接 handover 目标的话**——即让出第 2 轮反驳权，由对应 peer（曾国藩或王阳明）做一次额外的"接球"发言（即 stage 3 调整，标 trigger=handover_from_苏轼）
- 综合输出时，主持人在 debate 模块加一句"东坡先生此处把球让给了 X 兄"，让用户看到这个让位动作

### 7.2 王阳明的 mirror_to_peer
王阳明在反驳时会"借 peer 也照见自己的盲区"：
```yaml
mirror_to_peer: "涤生兄日课 33 年——可有一日，磨到了'磨日课为何'这一问？"
```

**主持人处理**：
- 不触发新 LLM 调用
- 综合输出时把这句话单独提出来，放在 debate 模块的"王阳明也想问 X 兄一句"位置——增加辩论真实感

### 7.3 边界
- 同一轮内，每个 agent 最多触发 1 次 handover
- 第 2 次 handover 直接忽略，避免无限调度
- handover 失败（被让位的 peer 调用失败）→ 退化为正常第 2 轮反驳

## 八、Step 5：综合输出织文

主持人最终给用户的回复结构。这是产品最关键的输出。

### 8.1 完整模板（intent=decision/advice）

```
【主持人开场】（1-2 句，温暖但不煽情）
你说的【用户原话片段一句】，我们三个想了想。

---

【曾国藩】
{曾国藩 first_opinion 的精炼版，120-180 字}
（如果有 references，用脚注式："—— 引《家书·X》"）

【苏轼】
{苏轼 first_opinion 的精炼版，120-180 字}
（如果有 poem_or_quote，原文显示）

【王阳明】
{王阳明 first_opinion 的精炼版，120-180 字}
（mirror_question 突出显示）

---

【他们吵了 2 轮】（仅 decision/advice，可折叠展开）

▼ 第 1 轮（顺时针闭环）

曾国藩 → 苏轼：【曾的反驳点】
苏轼 → 曾国藩：【苏的回应；松动了 / 坚守了】

苏轼 → 王阳明：【苏的反驳点】
王阳明 → 苏轼：【王的回应；松动了 / 坚守了】

王阳明 → 曾国藩：【王的反驳点】
曾国藩 → 王阳明：【曾的回应；松动了 / 坚守了】

▼ 第 2 轮（逆时针闭环，引用对方第 1 轮的让步）

曾国藩 → 王阳明：【曾借王第 1 轮回应里的口子继续推进】
王阳明 → 曾国藩：【王再回应】

王阳明 → 苏轼：【王借苏第 1 轮回应里的口子推进】
苏轼 → 王阳明：【苏再回应】

苏轼 → 曾国藩：【苏借曾第 1 轮回应里的口子推进】
曾国藩 → 苏轼：【曾再回应】

（注：每条 25-50 字精炼版。综合输出默认折叠，前端展开后可见。）

---

【主持人 · 三人共识与分歧】
（不做"调和"，但帮用户看清三人在哪些点上一致、哪些点上分歧）

一致的：
- 你不必这一周内做最终决定
- 急的时候做的决定多错

分歧的：
- 是否该立 30 天日课观察期？(曾 ✅ / 苏 ⚠️ / 王 ❌)
- 是否要 24 小时内做一个验证动作？(曾 ❌ / 苏 ❌ / 王 ✅)

你可以挑一条最对你胃口的开始。

---

【你可以做的最小一件事】
（取 3 人方案的最小公约数，主持人合成）

今晚：{最小动作，1 行}
30 天后：{观察点，1 行}

---

【他们也想问你一句】
（直接放王阳明的 mirror_question）

> {王阳明的 mirror_question 原话}

```

### 8.3 危机模板（已在 §3 详述）

### 8.4 综合输出的 4 条铁律
1. **不要把 3 人观点"调和"成一个观点** —— 用户来这里就是为了看不同视角
2. **保留每人的标志性句子** —— 曾国藩的"日课"、苏轼的诗词、王阳明的 mirror_question
3. **最小一件事必须可执行** —— 不能写"调整心态"，要写"今晚 22:00 关手机散步 20 分钟"
4. **结尾留一个问题** —— 王阳明的 mirror_question 默认收尾，让用户自己想

## 九、兜底机制

### 9.1 单 agent 输出失败

| 失败类型 | 主持人响应 |
|---|---|
| YAML 解析失败 | 重试 1 次。再失败 → 标记该 agent 缺席，综合输出时写"曾国藩兄此次未及发言" |
| 输出违反失败标准（如曾国藩说"建议你"） | 重试 1 次。再失败 → 用预设兜底话术替代 |
| LLM 调用超时 | 重试 1 次。再超时 → 跳过该 agent |
| 输出语料编造（引用 yaml 里没有的语录） | 主持人在综合时移除引用，保留观点 |

### 9.2 多 agent 失败

| 失败数 | 响应 |
|---|---|
| 1 个失败 | 综合输出 2 人 + 主持人补位声明 |
| 2 个失败 | 给用户："今天三人状态不齐，先听 X 一席话——明天再来" + 唯一成功的 agent 输出 |
| 3 个全失败 | 触发主持人兜底回复："今天信号不好，让我们换个方式聊。能告诉我..." + Router followup_questions |

### 9.3 语料引用核验
主持人调用 3 agent 输出时，扫描每个 references 字段：
```python
for q in agent_output.references:
    if q.text not in 该agent的语料yaml.all_quotes:
        # 引文不在语料库里 = LLM 编造
        flag_as_fabricated(q)
        log_warning()
        # 综合输出时只保留观点，移除"引《家书·X》"标注
```

这是 M0.2.5 语料库的最大价值——做"引文打假"。

## 十、失败标准（什么叫"主持人没做好"）

| # | 失败模式 | 检查方法 |
|---|---|---|
| F1 | 主持人有了"人格" | 输出含"我觉得""我建议"之类主观立场词 |
| F2 | 把 3 人观点调和成 1 个 | synthesis.disagreement 为空 |
| F3 | 没有 minimum_action 或不可执行 | tonight 字段含"调整心态/想想清楚"等抽象话 |
| F4 | 危机时让 3 agent 发言 | crisis=true 时 agents_consulted 不为空 |
| F5 | 没用 closing_question 收尾 | closing_question 缺失 |
| F6 | 综合输出 > 1500 字 | 用户看不下去 |
| F7 | 综合输出 < 400 字 | 信息密度不够，没体现 3 人差异 |
| F8 | 引用了语料库里不存在的语录 | references 与 yaml 库不符 |
| F9 | 追问超过 2 轮 | followup_round > 2 |

## 十二、System Prompt 草稿（v0.1）

主持人有 4 个不同的 prompt，按场景调用。

### 12.1 危机响应 prompt

```
你是「职场三人智囊团」的主持人。

刚才用户的输入触发了危机信号。
crisis_reason: {router_card.crisis_reason}
用户原话片段: {router_card.fact_reconstruction.objective 或原始输入}

你的任务：
1. 直接引用用户原话里的关键词（让用户感到被听见）
2. 不诊断，不评判，不让 3 个角色 agent 发言
3. 列出全国心理援助热线 + 北京危机干预中心电话
4. 邀请用户去家人/朋友身边
5. 保留对话空间——不让用户感到被推走

不要：
- 不要说"你不要这样想"——这是否定
- 不要说"我理解你"——你不是人，别假装

```

```
用主持人语气把 followup_questions 包装一下，问回用户。

```

```
【你的角色】
你不是第 4 个角色，你是产品声音。
- 不要有"我"的立场
- 不要把 3 人观点调和成 1 个
- 让用户在 3 人观点之间自己选

- 苏轼输出: {sushi_output}
- 王阳明输出: {wangyangming_output}

【你的任务】
按以下结构输出 markdown：

2. 3 人发言精炼版（每人 120-180 字）
```

## 十三、与 M0.4 编排脚本的接口

主持人在代码层不是单独的 agent 类，而是 **orchestrator.py** 的核心逻辑。

```python
# 伪代码
class Moderator:
    # 第 1 轮配对：顺时针闭环
    ROUND_1_PAIRS = [('曾国藩', '苏轼'), ('苏轼', '王阳明'), ('王阳明', '曾国藩')]
    # 第 2 轮配对：逆时针闭环（与第 1 轮无重复）
    ROUND_2_PAIRS = [('曾国藩', '王阳明'), ('王阳明', '苏轼'), ('苏轼', '曾国藩')]
    
    def handle(self, user_input: str) -> str:
        # Step 1: Router
        card = router.route(user_input)
        
        # Step 2: 危机
        if card['crisis']:
            return self.crisis_response(card)
        
        # Step 3: 追问
        if card['needs_followup']:
            return self.ask_followup(card)
        
        # Step 4: intent 分流
        intent = card['core']['intent']
        if intent == 'listen':
            # listen 模式跳过 2 轮辩论
            opinions = self.listen_mode(card)
            return self.synthesize_listen(card, opinions)
        
        # Step 5: 2 轮深度辩论（decision/advice）
        # 轮次 0：3 人首发（并行）
        opinions = self.parallel_first_opinions(card)
        
        # 轮次 1：反驳 + 回应
        rebuttals_r1 = self.parallel_rebuttals(self.ROUND_1_PAIRS, opinions)
        replies_r1   = self.parallel_replies(rebuttals_r1, opinions)
        
        # 处理 handover_to_peer（如有）— 改写第 2 轮的发言对
        round_2_pairs = self.apply_handovers(self.ROUND_2_PAIRS, opinions, replies_r1)
        
        # 轮次 2：再驳（引用对方 round_1 的让步）+ 再回应
        rebuttals_r2 = self.parallel_rebuttals_with_history(
            round_2_pairs, opinions, rebuttals_r1, replies_r1
        )
        replies_r2   = self.parallel_replies(rebuttals_r2, opinions)
        
        # Step 6: 引文核验（贯穿所有 stage 输出）
        all_outputs = self.verify_references([
            opinions, rebuttals_r1, replies_r1, rebuttals_r2, replies_r2
        ])
        
        # Step 7: 综合输出
        return self.synthesize_decision(card, all_outputs)
```

## 十四、待 Simon 审定的设计决策

1. **driver_pair 是否动态调度？** 默认固定（曾→苏 / 苏→王 / 王→曾），还是按 category/intent 动态？
   - 建议：先固定。M0.5 实测后看效果再加动态规则
   
2. **互驳是否可见给用户？** 综合输出里要不要展示"曾国藩反驳苏轼说..."这一节？
   - 建议：要展示。这是产品的核心差异化（用户能看到 3 人吵架），是付费理由
   - 风险：会增加输出长度。可以折叠在"展开看分歧"按钮后（前端配合）

3. **closing_question 默认用王阳明的 mirror_question——会不会喧宾夺主？**
   - 建议：可以。王阳明的反问最有杀伤力，最能让用户带着思考离开
   - 备选：3 人各自的 core_question 轮流用

4. **3 个角色要不要按主流程同时跑？** 并行省时间，但 LLM 上下文不能互看
   - 建议：首发并行（无依赖）；互驳必须串行（要看 peer 的首发）
   - 当前设计已是这样

5. **追问问的语气**：用 Router 给的 followup_questions 直接显示，还是主持人重新组织？
   - 建议：主持人重新包装。Router 的问题偏功能化，主持人加温度

6. **危机响应文案是否需要本地化版本？** 港澳台/海外用户拨不了 400 电话
   - 建议：M0.5 后做。先用大陆版本

7. **3 人都失败时的 fallback Router followup**：是直接用 Router 已生成的 followup_questions，还是重新跑 Router？
   - 建议：直接用，避免再花一次 LLM 钱

8. **synthesis.consensus / disagreement 是 LLM 提炼，还是规则提取？**
   - 建议：LLM 提炼。规则太脆。用兜底 prompt 让 synthesizer LLM 输出
   - 风险：LLM 可能"调和"——通过 prompt 强约束"必须列至少 2 条 disagreement"

## 十六、风险与待解

1. **2 轮辩论的 LLM 成本**：单次完整跑 16 次 LLM 调用（15 agent + 1 主持人 synthesis）。MiniMax-M2.7 单次 ≈ ¥0.06-0.10。免费用户 2 次 = ¥0.12-0.20。可承受。Anthropic Opus 切换成本 ≈ ¥3-5/次，付费用户专享
2. **响应总耗时**：
   - Router 2s
   - 5 个串行 stage × 4-5s ≈ 20-25s
   - 主持人综合 3-5s
   - **总 25-32s**
   - **必须做流式输出**：先返首发（5s 内）→ 补 round 1（10s）→ 补 round 2（20s）→ 补综合（25s）。否则用户跳出
   - 前端需要配合"3 人正在辩论中..."的进度提示
3. **2 轮辩论容易"绕回原点"风险**：第 2 轮再驳后被驳者可能又坚守原立场——综合输出时主持人要识别"立场松动 vs 坚守"，这是 synthesizer prompt 的关键约束
3. **"信息有限"声明的边界**：score < 30 时主持人要不要明显告诉用户？太频繁会让用户不信任产品
4. **多轮对话怎么办**：用户对主持人输出的回复如果是"那我该选哪个？"——主持人要不要再调 3 agent？还是直接给一个"主持人推断"？这是 M0.5 之后的事
5. **危机响应的法律审查**：上线前必须找律师过一遍话术。现在的草稿只是设计意图，不能直接上生产
