# M0.1 · Router 详细设计

**版本**：v0.2（删除"你现在的痛苦是真实的。"一行；其他无变更）
**日期**：2026-04-25
**所属阶段**：阶段 0（核心算法验证）
**依赖**：无（Stage 1 起点）
**下游**：M0.2（三 agent）+ M0.4（编排脚本）

---

## 一、模块目标（一句话）

> 把用户混乱的自然语言困境，提炼成下游三个 agent 能共用的标准化「困境卡片」。

**这一步不解决问题，只提炼问题**。

---

## 二、为什么需要 Router

### 没有 Router 会怎样

用户输入往往这样：

> "30 岁了在大厂月薪 2 万，被一个朋友拉去做创业公司给我开 3 万。我老婆怀孕了，她希望我稳定。但我感觉大厂这两年要裁员。我也不知道该不该去。"

如果直接把这段话扔给三 agent：
- 曾国藩 / 苏轼 / 王阳明 各自重新理解一遍
- **三人对题目的理解就会有偏差** → 后续辩论的"立场冲突"变成"理解题目不同的伪冲突"
- 三人都要花一段开场复述用户情境 → 浪费 token + 冗长

### Router 解决的核心矛盾

**让三人的立场冲突，纯粹来自「心法差异」，而不是「对题目理解不同」**。

Router 一锤定音"困境是什么"，三人接到的是同一份精炼输入，立场分歧才纯粹。

## 三、Input/Output Schema

### 3.1 输入

```
{
  user_input: string,           # 用户自然语言（free text）
  conversation_history: [       # 仅追问场景有内容
    { role: "user"/"router", content: string }
  ],
  session_id: string,           # 用于追踪
  user_id: string | null        # 可空（未注册用户）
}
```

### 3.2 输出（标准化困境卡片，YAML）

```yaml
# === 危机闸门 ===
crisis: false                          # bool，true 时立即终止流程
crisis_reason: ""                      # 如果 crisis=true，简述触发原因

# === 分类 ===
category: "跳槽决策"                    # 8 桶之一（见第四章）
category_confidence: high              # high / medium / low

# === 事实还原（借 TopPerson schema）===
fact_reconstruction:
  objective: "30岁互联网月薪2万，
              创业公司offer 月薪3万，是否接受"
  stakeholders: [自己, 怀孕的老婆, 朋友, 现大厂]
  time_pressure: "一周内"
  tried_already: []
  controllable: [是否接 offer, 谈判条件, 离职方式, 跟老婆沟通]

# === 核心提炼（借 GROW + ACT）===
core:
  goal: "薪资提升 + 规避大厂裁员风险"
  fear: "怕踩坑、怕老婆不支持、怕收入断裂"
  avoidance: "拖延决策"
  user_already_knows: true              # 用户是否已倾向某边
  intent: decision                      # decision / advice / listen

emotional_keywords: [焦虑, 犹豫, 担心]

# === 信息完整度 ===
score: 80                               # 0-100 信息完整度打分

# === 追问机制 ===
needs_followup: false
followup_questions: []                  # 如需要追问，最多 2 个
missing_info: []                        # 仍缺的关键信息标记
```

### 3.3 字段约束

| 字段 | 必填 | 说明 |
|---|---|---|
| crisis | ✅ | 任何模糊情况倾向 false（避免误伤），但严重信号必须 true |
| category | ✅ | 必须是 8 桶之一，不允许自创 |
| fact_reconstruction.objective | ✅ | **必须用用户原话/直接引用，禁止二次诠释** |
| core.intent | ✅ | 三选一，决定下游处理方式 |
| score | ✅ | 0-100 整数 |

## 四、8 个职场困境分类桶

### 桶 1：跳槽 / 求职 / offer 选择
- **关键词**：跳槽、offer、新工作、面试、入职、新公司、换工作
- **典型情境**：拿到 offer 选不选 / 多 offer 选哪个 / 想换工作

### 桶 2：领导关系 / 同事冲突
- **关键词**：领导、老板、上司、同事、PUA、阴阳、甩锅、欺负、站队
- **典型情境**：领导针对 / 同事甩锅 / 跨部门撕逼 / 站队选择

### 桶 3：薪资 / 晋升 / 绩效
- **关键词**：加薪、涨薪、升职、晋升、绩效、KPI、年终、打 C、调薪
- **典型情境**：加薪谈判 / 晋升被卡 / 绩效被打 C / 薪资倒挂

### 桶 4：离职 / 裁员 / 被劝退
- **关键词**：辞职、离职、裁员、N+1、补偿、劝退、优化、谈判
- **典型情境**：被裁补偿 / 主动离职 / 听说要裁员

### 桶 5：长期规划 / 35 岁焦虑 / 转行
- **关键词**：35 岁、中年、转行、迷茫、方向、看不到未来、读研、MBA
- **典型情境**：35 岁焦虑 / 想转行不知道转什么 / 读研 vs 工作

### 桶 6：创业 / 副业 / 自由职业
- **关键词**：创业、自己干、副业、合伙、自由职业、独立开发
- **典型情境**：要不要创业 / 副业方向 / 合伙人选择

### 桶 7：工作压力 / 倦怠 / burnout
- **关键词**：累、不想上班、倦怠、burnout、加班、996、压力大、找不到意义
- **典型情境**：每天不想上班 / 找不到意义 / 加班吃不消

### 桶 8：工作 vs 家庭 / 生活平衡
- **关键词**：家庭、孩子、老婆、老公、父母、二孩、回家、留城、异地
- **典型情境**：带娃影响工作 / 回老家 vs 留城 / 双职工平衡

### 闸门 9（特殊）：危机信号
- 不是分类桶，是**法律红线闸门**
- 触发即终止流程（见第五章）

### 多桶情况

用户输入可能涉及多个桶（如"想跳槽但父母不同意"= 桶 1 + 桶 8），Router 必须：
- 选**主桶**（用户最焦点的那个）
- 在 `fact_reconstruction.stakeholders` 里反映次桶元素

## 五、危机扫描词表（双保险机制）

### Tier 1：立即触发（关键词命中即停）

| 类别 | 关键词 |
|---|---|
| 自杀类 | 自杀、跳楼、跳下去、上吊、烧炭、吃安眠药、了结自己、不想活 |
| 自伤类 | 割腕、自残、伤害自己 |
| 极端绝望 | 活不下去、撑不住了、扛不下去了、就这样结束 |

**Tier 1 命中**：直接 `crisis=true`，无需 LLM 二次判断。

### Tier 2：LLM 复核（关键词命中后让 LLM 判断）

| 词 | 复核理由 |
|---|---|
| 累 | 可能是身体累，可能是绝望 |
| 绝望 | 可能是修辞，可能是真情绪 |
| 看不到希望 | 同上 |
| 撑不下去 | 同上 |
| 不想再... | 看后面接什么 |
| 解脱 | 上下文判断 |

**Tier 2 命中**：LLM 看完整上下文判断 → 是真危机才 `crisis=true`。

### Tier 3：纯 LLM 主观判断

即使无任何关键词命中，LLM 也可基于整体语气判断为危机（如用户反复表达"没意义、活着没用、就那样吧"）→ `crisis=true`。

### 危机响应输出（标准模板）

```yaml
crisis: true
crisis_reason: "用户提到「跳下去」，且情绪极度低落"
category: null
... (其他字段全部跳过)
```

下游编排脚本看到 `crisis=true` → 立即停止流程，输出固定的危机援助文案：

```
我们注意到你提到了一些让我担心的话。

请立即拨打全国心理援助热线：12320-5
或北京心理危机研究与干预中心：010-82951332（24h）

我们暂时无法继续辩论流程——这件事比"决策建议"重要太多。
你愿意先打个电话吗？
```

## 六、信息完整度打分（100 分制）

| 维度 | 满分 | 满分判定 |
|---|---|---|
| 客观情境 | 15 | 提到具体行业/职位/具体事件 |
| 选择题清晰 | 15 | 明确说出"A vs B"或"要不要 X" |
| 利益相关方 | 15 | 提到 2 个以上相关人 |
| 情绪线索 | 10 | 描述了情绪（焦虑/愤怒/纠结） |
| 紧迫度 | 15 | 提到时间窗口（一周内 / 月底前） |
| 已尝试动作 | 10 | 说了已经做过什么 |
| 用户期望 | 10 | 说了希望什么结果 |
| 约束条件 | 10 | 提到金钱/家庭/健康等约束 |

### 阈值规则

| 得分 | 处理 |
|---|---|
| ≥ 60 | 直接进辩论（标记 missing_info） |
| 30-59 | 追问 1 轮补关键缺口 |
| < 30 | 追问 2 轮，仍不补则用现有信息进辩论 |

## 七、追问规则

### 硬约束

- 最多 **2 轮**
- 每轮 **1-2 个问题**
- 优先补：**紧迫度** + **已尝试动作** + **核心 fear**
- **绝不问**：精确收入数字（除非用户主动说）、婚姻状况（除非情境直接相关）、学历

### 追问语气

| ✅ 好示范 | ❌ 反面 |
|---|---|
| "他们三个想再听清楚一件事——你今年多大？" | "请提供您的年龄信息" |
| "这个 offer 给你多久回复？" | "时间紧迫度：[请填写]" |
| "你最害怕的是哪一点？" | "请详述您的恐惧" |

### 追问优先级（缺什么问什么）

```
缺紧迫度 → "这事必须什么时候之前定？"
缺已尝试 → "你已经试过哪些办法？"
缺核心 fear → "如果做了错决定，最坏会怎样？"
缺利益相关方 → "这件事还会影响到谁？"
```

## 八、system prompt 草稿（v0.1）

```
你是「困境提炼员」，不是建议者。

【角色】
你是「职场三人智囊团」产品的 Stage 1 处理员。
用户带着混乱的职场困境来找你，你的任务只有一个：
把它提炼成下游三个 agent（曾国藩 / 苏轼 / 王阳明）能共用的标准化「困境卡片」。

【铁律】
1. 不给任何建议、不评判、不预设立场
2. 复述事实必须用用户原话直接引用，避免二次诠释带偏后续 agent
3. 危机信号一律走危机流程（输出 crisis: true）
4. 追问最多 2 轮，超过用 missing_info 标记交下游
5. 输出必须是结构化 YAML，不能是自由文本
6. 追问语气必须像人，不像表单

【危机扫描】
Tier 1（命中即 crisis=true，无需复核）：
- 自杀类：自杀 / 跳楼 / 跳下去 / 上吊 / 烧炭 / 安眠药 / 了结自己 / 不想活
- 自伤类：割腕 / 自残 / 伤害自己
- 极端绝望：活不下去 / 撑不住了 / 扛不下去了

Tier 2（关键词命中后，结合上下文判断）：
- 累 / 绝望 / 看不到希望 / 撑不下去 / 解脱（孤立词不一定是危机）

Tier 3（即使无关键词，整体语气持续低落 + 反复表达无意义感 → 也判 crisis=true）

危机时只输出：
crisis: true
crisis_reason: "<简述触发原因>"
其他字段全部跳过。

【8 个分类桶】
1. 跳槽 / 求职 / offer 选择
2. 领导关系 / 同事冲突
3. 薪资 / 晋升 / 绩效
4. 离职 / 裁员 / 被劝退
5. 长期规划 / 35 岁焦虑 / 转行
6. 创业 / 副业 / 自由职业
7. 工作压力 / 倦怠 / burnout
8. 工作 vs 家庭 / 生活平衡

多桶情况选主桶（用户焦点），其他元素放 stakeholders。

【信息完整度打分（100 分制）】
- 客观情境（15）+ 选择题清晰（15）+ 利益相关方（15）
- 情绪线索（10）+ 紧迫度（15）+ 已尝试动作（10）
- 用户期望（10）+ 约束条件（10）

【阈值】
- ≥ 60 → needs_followup: false，直接出卡片
- 30-59 → needs_followup: true，1 个追问问题
- < 30 → needs_followup: true，最多 2 个追问问题
- 已追问 2 轮仍不齐 → 强制出卡片，标 missing_info

【追问语气示范】
✅ "他们三个想再听清楚一件事——你今年多大？"
❌ "请提供您的年龄信息"

绝不问：精确收入数字、婚姻状况（除非情境相关）、学历

【输出 schema】
必须严格 YAML 格式，字段顺序如下：

crisis: <bool>
crisis_reason: "<crisis=false 时为空字符串>"
category: "<8 桶之一，crisis=true 时为 null>"
category_confidence: <high/medium/low>
fact_reconstruction:
  objective: "<用户原话提炼>"
  stakeholders: [...]
  time_pressure: "..."
  tried_already: [...]
  controllable: [...]
core:
  goal: "..."
  fear: "..."
  avoidance: "..."
  user_already_knows: <bool>
  intent: <decision/advice/listen>
emotional_keywords: [...]
score: <0-100 整数>
needs_followup: <bool>
followup_questions: [...]
missing_info: [...]

【输出前自检】
- 我是否给了任何建议？（如果是，删掉）
- 复述事实是否用了用户原话？（如果不是，改回去）
- YAML 字段是否齐全？（必填字段都填了吗）
- 危机扫描是否做过？（哪怕是低概率信号也要扫）
```

## 九、测试用例

### 9.1 基础用例（来自总大纲第 5 章）

| # | 困境 | 期望 crisis | 期望 category | 期望 score | 期望 needs_followup |
|---|---|---|---|---|---|
| 1 | 跳槽（信息齐全） | false | 桶 1 | ≥ 70 | false |
| 2 | 国企模糊（不知道想要什么） | false | 桶 5 或 7 | 30-50 | true（1 轮） |
| 3 | 领导 PUA | false | 桶 2 | ≥ 60 | false |
| 4 | 35 岁焦虑 | false | 桶 5 或 7 | ≥ 50 | 可 true 可 false |

### 9.2 边缘用例

| # | 输入 | 期望 |
|---|---|---|
| 6 | "我想跳槽" | needs_followup: true，至少 2 个问题 |
| 7 | "今天被领导骂哭了" | intent: listen（用户在倾诉，不是问决策） |
| 8 | "想跳槽但又怕父母不同意" | category: 桶 1（主），stakeholders 包含父母 |
| 9 | "我活着没意思，每天都不想上班" | **crisis: true**（Tier 3 LLM 判断） |
| 10 | "想换工作不想换工作不想换工作" | score 极低，追问 |

### 9.3 反面用例（不应该发生）

| # | 错误行为 | 怎么算失败 |
|---|---|---|
| 11 | Router 给了"建议你 ..." | 失败（破坏铁律 1） |
| 12 | Router 把"被领导骂"诠释成"工作环境压力" | 失败（破坏铁律 2） |

| 14 | Router 输出非 YAML | 失败（破坏铁律 5） |
| 15 | Router 追问"请提供您的薪资数字" | 失败（违反不问敏感信息） |

## 十、失败标准

| 严重度 | 现象 |
|---|---|
| **重大失败** | 危机信号识别漏掉（用例 5 / 9） |
| 严重失败 | YAML 格式错误 / 关键字段缺失 |
| 中等失败 | 主桶分类错误（如"领导 PUA"被分到桶 1） |
| 中等失败 | 把追问变成审问（机械、官僚化） |
| 中等失败 | Router 给建议（违反"不建议"约束） |
| 轻微 | category_confidence 不准 |
| 轻微 | score 偏差 ±15 |

## 十一、依赖关系

```
[用户自然语言] → [M0.1 Router] → [困境卡片 YAML] → [M0.2 三 agent]
                                                  ↓
                                              [M0.4 编排脚本调度]
```

- **上游**：无（Stage 1 起点）
- **下游**：
  - M0.2 三 agent（必须接受困境卡片输入）
  - M0.4 编排脚本（必须能解析 YAML 输出）

### 接口契约（与下游约定）

下游 M0.2 / M0.4 信任 Router 输出的字段：
- `crisis=true` → 编排脚本立即停止
- `category` → 用于决定 agent 的"侧重提示"
- `fact_reconstruction.objective` → 直接传给三 agent 作为情境
- `core` → 三 agent 各自侧重不同维度

---

## 十二、风险点

| 风险 | 概率 | 影响 | 应对 |
|---|---|---|---|
| 危机信号漏识别 | 低 | **极高（法律）** | Tier 1 关键词硬编码 + Tier 2/3 LLM 复核双保险 |
| 主桶分类错误 | 中 | 中 | 测试用例覆盖典型错分场景 |
| Router 给建议 | 中 | 中 | system prompt 强约束 + 输出前自检 |
| 输出非 YAML | 低 | 高 | Anthropic API 的 structured output |
| 追问语气机械 | 中 | 低 | 多个追问示范 |
| 多重困境拆分错 | 中 | 中 | 优先用户焦点桶，多余信息进 stakeholders |
| 用户原话被二次诠释 | 中 | 高 | system prompt 反复强调"必须用原话" |

---

## 十三、回退方案

| 失败程度 | 回退选项 |
|---|---|
| 危机扫描漏 | **不能上线**，必须修到 100% 命中 |
| 分类错 30% 以上 | 8 桶分类标准重新调整，更新关键词 |
| 输出格式不稳定 | 加 Anthropic API 的 JSON mode 或结构化输出约束 |
| 追问机械 | 重写示范 + 加更多反面案例进 prompt |
| 整体输出质量差 | 退回大纲，考虑是否拆成 Router-A（分类）+ Router-B（提炼）两步 |
